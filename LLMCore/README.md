# **LLMCore 设计概述**

本框架旨在提供一个灵活、可扩展的大模型模块，支持多种场景的开发，如游戏、文档对话助手，客服等。框架通过模块化设计，使每个功能模块独立工作，并且通过插件机制，支持开发者对核心功能进行扩展和定制。以下设计中的功能说明将以游戏为例，但理论上这些功能可以扩展到其他使用场景。

## **1. 身份与对话规则模块**

定义和管理大模型的身份及其交流规则；
设定不同的使用场景（例如游戏中的DND/狼人杀主持人角色、项目文档对话助手，企业客服等）；
设定大模型的身份属性（如名称、语气、交流规则等）；
利用 LangChain 生成对话模板，提供预设，并提供修改支持。

### **使用场景设定**

开发者可以选择不同的预设使用场景，如“游戏”，“客服”等，不同场景支持的后续设定可能会有所不同。

### **身份属性设定**

开发者可以定义AI的身份属性，包括但不限于：

1. 名称（如“Dungeon Master”或“客服小智”等）
2. 语气（如“严肃”或“友好”等）
3. 行为特征（如是否主动提供建议、是否引导用户思考等）
4. 人设性格特点（如，你是一只猫娘，你有幽默感、耐心等）
5. 语言风格（使用正式、非正式、专业术语或简单易懂的语言等）

### **对话规则定义**

允许设定AI与用户的互动规则，包括但不限于：

1. 提示频率（游戏场景中控制模型主动提供帮助或提示的频率）；
2. 对话轮次（设定每次互动的最大轮次数，以避免对话过长或过短）；
3. 用户输入处理（定义如何处理用户输入，如是否允许自由输入、是否提供输入建议或限制输入范围）；
4. 互动模式（启发式提问、指令式对话、开放式讨论等）；
5. 容错性（设定模型在处理用户不明确或错误输入时的应对策略，如重述、提供选项或直接跳过）；

### **模板生成**

根据使用场景和设定的属性，使用 LangChain 的 `PromptTemplate` 生成对应的对话模板。

### **扩展性**

- 支持预设场景的扩展和对应场景相关设定API的注册；
- 支持身份属性的扩展，例如支持设定屏蔽词，使用语种等；
- 支持对话规则的扩展，例如自行设计的策略等；
- 支持模板预设的增加。

## **2. 背景规则模块**

管理和应用场景的背景规则。根据不同的使用场景提供规则和机制的定义方法。

### **规则定义**

开发者可以为特定场景定义背景规则，包括但不限于：

1. 定义场景的背景信息，如游戏中的故事设定、历史背景，或企业场景中的产品定位说明、法律条款等；
2. 定义场景中的操作流程规则，如游戏中的任务目标、关卡流程，或企业中的服务流程、用户操作指导等；
3. 场景特性，如时间线、环境设定、事件触发条件等。

### **背景整合**

1. 利用 LangChain，将背景规则模板化，通过 `PromptTemplate` 生成适合当前场景的背景描述。
2. 预设的长文可以使用 RAG 当作背景知识库。
3. 背景信息将被自动整合到对话中，影响模型的生成结果。

### **扩展性**

- 允许开发者扩展背景规则，支持新的背景信息类型或复杂的背景设定；
- 允许集成新的数据源或知识库管理工具；
- 支持开发者创建和管理新的背景模板，以适应不同的使用场景和需求；

## **3. 动态知识库模块**

利用 LangChain 和 RAG 技术，动态管理场景中的动态知识库，记录并更新场景过程中产生的重要信息。这包括对用户操作的记忆、知识库的动态更新，以及支持场景状态的回退和重置。

### **关键步骤记忆**

开发者可以定义什么是关键步骤的 Triger，一旦触发 Triger，关键步骤将更新到知识库中，记录用户执行了这一操作，例如：

1. 在游戏场景中，当玩家完成重要任务时，触发更新；
2. 在企业对话中，当客户完成关键操作（如确认订单）时，记录该操作；

### **知识库更新**

开发者可以在场景过程中动态更新知识库内容，确保知识库能够反映当前状态，供后续检索，例如：

1. 在游戏中更新故事情节，例如某一轮结束，解锁新关卡/地图，用户关键操作引发的新剧情；
2. 在企业对话中更新客户需求状态；

### **状态管理**

允许开发者在需要重置或回退时，恢复知识库到初始状态或特定步骤，例如：

1. 在游戏中回退到特定剧情节点；
2. 在企业对话中恢复到最初的需求状态；

### **扩展性**

- 允许扩展动态知识库的结构，支持更复杂的数据类型或定制的存储方式。
- 支持自定义的触发条件和更复杂的触发逻辑，以适应不同的场景需求。
- 支持更细粒度的状态回退和恢复策略，以应对更复杂的场景变化。

## **4. 对话流管理模块（不是很懂，GPT写的）**

### **对话上下文管理**

管理对话的上下文信息，确保模型在生成对话内容时能够正确引用之前的对话信息，保持对话的连续性和逻辑一致性。

**示例**：在游戏场景中，模型能够记住玩家之前的选择并在后续对话中引用；在企业对话中，模型能够记住客户之前提到的问题并基于此进行进一步交流。

### **对话状态追踪**

追踪对话的状态，包括当前对话的进展、已处理的内容、未解决的问题等。该功能确保对话能够根据用户的反馈进行调整，并避免重复或遗漏重要信息。

**示例**：在游戏中，跟踪玩家的对话选项，以确保后续的情节发展符合逻辑；在企业对话中，跟踪客户的需求状态，以便在后续交互中提供有针对性的服务。

### **动态对话策略调整**

根据用户输入的变化和对话的进展，动态调整对话策略，确保互动的有效性和用户体验的优化。例如，可以在对话的某个阶段切换交互模式或调整对话的语气。

**示例**：在游戏中，根据玩家的行为调整对话的难度或引导方式；在企业对话中，根据客户的情绪和反馈调整服务策略。

### **扩展性**

- 支持对话上下文的自定义管理策略，如自定义上下文保留时长、不同场景的上下文切换机制等。
- 允许开发者扩展对话状态追踪的粒度和方式，如添加自定义的状态标签或追踪逻辑。
- 支持更多的动态对话策略和自定义策略调整逻辑，适应更复杂的交互场景和用户需求。

## **5. 插件模块**

管理框架中的第三方插件，支持插件的注册、加载、启用和禁用，确保插件能够在框架中协同工作。

### **插件注册与管理**

开发者可以注册、启用或禁用插件。插件通过配置文件或API接口进行注册，并由插件管理器控制其生命周期。

### **扩展点管理**

插件模块为各个核心功能模块提供扩展点（Hooks），插件可以在这些扩展点上插入自定义逻辑，扩展或修改框架的默认行为。

### **插件接口标准化**

所有插件必须实现该接口的基本方法，包括初始化、执行和清理等步骤。框架在适当的时候调用这些方法，确保插件的正常运行。
